Review and merge documentation changes

All documentation changes must pass a pull-request review before merging into the main branch. This process ensures accuracy, consistency, and full traceability across every release of the published output.

Overview

A pull request (PR) is the formal mechanism for proposing, discussing, and approving documentation changes before they enter main. Every PR triggers a CI build that compiles the Flare project and reports errors as a required status check. The reviewer inspects both the raw file diff and the rendered preview before approving.

Author pre-review checklist

Complete the following checks before you open a pull request. Addressing these items before review reduces the number of review cycles.

Check How to verify
Topic builds without Flare errors or warnings Run a local Flare build (Build > Build Primary) and review the Build Log
Markdown syntax is valid Run markdownlint (or equivalent) on all changed .md files
New topics are added to the TOC Open the .fltoc file and confirm the new topic node is present
Cross-references are not broken Check the Flare Analyzer report (Tools > Project Analysis > Analyzer)
Content follows the style guide Review the Microsoft Writing Style Guide and the project style sheet
Commit messages follow the project convention See Commit message guidelines

Open a pull request

  1. Push the feature branch to the remote repository.
  2. In the remote repository UI, select New pull request (GitHub) or Create merge request (GitLab).
  3. Set the base branch to main and the compare branch to your feature branch.
  4. Complete the pull-request description using the project PR template. Include:
    • A summary of the documentation change and the reason for the change.
    • A link to the related issue or task (if applicable).
    • A screenshot or link to the rendered preview (if the change affects layout or formatting).
  5. Assign at least one reviewer and add relevant labels (for example, content or hotfix).
  6. Select Create pull request.

Reviewer checklist

As a reviewer, verify the following before approving a pull request:

  • The content is factually accurate and complete for the stated scope.
  • Headings, steps, and tables follow Microsoft Writing Style Guide conventions.
  • Code samples are syntactically correct and use the correct language identifier in fenced code blocks.
  • All new topics are linked in the TOC and in at least one related-topics section.
  • No sensitive information (API keys, credentials, internal URLs) is included in the content.
  • The CI build status check is passing.

Note  Leave inline comments on specific lines for targeted feedback. Use a general PR comment for overall observations. Mark comments as Resolved after the author addresses them.

Resolve merge conflicts

Merge conflicts occur when two branches modify the same file in incompatible ways. To resolve conflicts on a feature branch:

  1. Fetch and merge the latest main into the feature branch: 
    git checkout docs/your-feature-branch
git fetch origin
git merge origin/main
  2. Open each conflicting file. Git marks conflicts with the following delimiters: 
    <<<<<<< HEAD
  (your changes)
=======
  (incoming changes from main)
>>>>>>> origin/main
  3. Edit the file to keep the correct content. Remove all conflict delimiter lines (<<<, ===, >>>).
  4. Stage the resolved files: 
    git add <resolved-file>
  5. Complete the merge commit: 
    git commit -m "Merge main into docs/your-feature-branch; resolve conflict in <file>"
  6. Push the updated branch and re-request review on the pull request.

Important  Flare .fltoc and .flprj files are XML and can produce complex merge conflicts. Coordinate with the project administrator before making structural changes to these files, and resolve conflicts carefully to avoid corrupting the project structure.

Merge and clean up

  1. After the pull request is approved and all status checks pass, select Merge pull request (or Merge).
  2. Use the Squash and merge strategy for feature branches to maintain a clean commit history on main.
  3. Delete the feature branch from the remote repository.
  4. Update your local repository: 
    git checkout main
git pull origin main
git branch -d docs/your-feature-branch
