Set up Git for a Flare project

Initialize a Git repository for your MadCap Flare project so that all topic files—including Markdown sources—are tracked under version control from day one.

Before you begin

Confirm that the following software is installed and configured on the author workstation:

  • Git 2.34 or later, available on the system PATH
  • MadCap Flare 2019 R2 or later
  • An account with write access to a remote repository on GitHub, GitLab, or Azure DevOps

Note  To verify the Git installation, open a terminal and run git --version. The output should display git version 2.34.x or later.

Create the remote repository

Create an empty repository on your Git hosting platform before you initialize locally. Do not add a README, .gitignore, or license file during creation—these are added in the steps that follow.

  1. Sign in to your Git hosting platform (GitHub, GitLab, or Azure DevOps).
  2. Create a new repository. Copy the HTTPS or SSH remote URL for use in a later step.

Initialize the local repository

Perform these steps in the root folder of the MadCap Flare project—the folder that contains the .flprj file.

  1. Open a terminal in the Flare project root folder.
  2. Initialize the local repository: 
    git init
  3. Create a .gitignore file to exclude Flare build artifacts (see Recommended .gitignore entries).
  4. Stage all project files: 
    git add .
  5. Create the initial commit: 
    git commit -m "Initial commit: MadCap Flare project"
  6. Link the remote repository: 
    git remote add origin <remote-url>
  7. Push to the remote and set the upstream tracking branch: 
    git push -u origin main

Note  If your default branch is named master, rename it before pushing: git branch -M main.

Recommended .gitignore entries

Add the following entries to your .gitignore file. These entries exclude Flare build output and temporary files that should not be tracked in version control.

Entry Reason
Output/ Generated HTML and PDF build output—rebuilt by the CI/CD pipeline on every merge
*.mcbackup Flare automatic backup files, not required in version control
.mclog Flare session log files
*.flbkp Flare project backup snapshots
*.bak Generic editor backup files

Important  Never commit the Output/ folder to the repository. Committing build output inflates repository size and creates misleading diffs in pull requests.

Configure branch protection

After pushing the initial commit, configure branch protection rules on the remote repository to prevent direct pushes to main:

  1. In the repository settings, open Branches (GitHub) or Repository > Protected branches (GitLab).
  2. Add a protection rule for the main branch.
  3. Enable Require a pull request before merging.
  4. Set Required approvals to at least 1.
  5. Enable Require status checks to pass before merging and select the CI build check.

Admin note  Branch protection rules apply to all contributors, including administrators. Use a hotfix branch process for urgent corrections rather than bypassing protection.

Verify the setup

Confirm that the repository is correctly initialized:

  1. Run git status. The output should display nothing to commit, working tree clean.
  2. Run git log --oneline. The initial commit should appear in the log.
  3. Run git remote -v. The remote URL should be listed under origin.
