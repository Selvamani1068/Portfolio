Developer documentation with Markdown and Git in MadCap Flare

Learn how to author, version, and publish developer documentation by using Markdown files and Git version control inside a MadCap Flare project. This guide covers project setup, authoring conventions, branching workflows, and automated publishing.

Overview

MadCap Flare supports Markdown (.md) topic files natively alongside XML-based .htm topics (Flare 2019 R2 and later). By combining Flare's single-source publishing engine with Git version control, documentation teams can:

  • Let developers contribute content in Markdown without learning Flare XML.
  • Track every change, review diffs, and attribute edits through Git history.
  • Compile Markdown and XML topics into a single, consistent HTML5 WebHelp output.
  • Trigger automated Flare builds on every merge by using a CI/CD pipeline.

Prerequisites

Before you begin, make sure the following requirements are met:

Requirement Minimum version Notes
MadCap Flare 2019 R2 (v15.1) Required for native Markdown topic support
Git 2.34 Must be available on the system PATH
Remote repository Any GitHub, GitLab, or Azure DevOps
Author permissions — Write access to the target repository branch

Supported outputs

WebHelp: Responsive HTML5 output that is optimized for desktop and tablet browsers. This is the primary delivery channel for developer portal documentation.

Output type Format Notes
Online help HTML5 WebHelp Primary delivery channel for developer portals
PDF guide PDF Generated from the same source by using Flare print targets
Context-sensitive help CSH IDs Linked from IDE or CLI tools by using help buttons

In this guide

This guide is organized into the following sections:

  • Setting up Git — Initialize a repository and configure the recommended .gitignore entries.
  • Markdown in Flare — Create Markdown topic files and understand supported syntax elements.
  • Linking Flare and Git — Bind the Flare project to a remote repository by using the built-in Source Control integration.
  • Branching strategy — Follow a consistent branching model to keep review and release workflows separate.
  • Review and merge — Conduct pull-request reviews and resolve merge conflicts.
  • Publishing and CI/CD — Automate Flare builds and deploy the output on every merge.
