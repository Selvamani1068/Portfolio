Create and manage developer documentation using Markdown and Git

This guide shows you how to write developer documentation in Markdown, manage it with Git version control, and publish it from MadCap Flare. Each section contains step-by-step procedures that follow the Microsoft Writing Style Guide.

What you will do

By following this guide, you will:

  • Set up a Git repository for your documentation project.
  • Create and edit Markdown topic files inside MadCap Flare.
  • Connect your Flare project to a remote GitHub repository.
  • Use a branching strategy to manage content changes safely.
  • Submit, review, and merge documentation changes through pull requests.
  • Automate Flare builds and publish documentation with a CI/CD pipeline.

Who this guide is for

This guide is intended for technical writers and developer-documentation authors who:

  • Use MadCap Flare as their authoring tool.
  • Want to write or accept contributions in Markdown format.
  • Need to version-control documentation alongside software source code.
  • Want to automate the build and deployment of their documentation output.

Prerequisites

Before you begin, make sure the following requirements are met:

Requirement Minimum version Notes
MadCap Flare 2019 R2 (v15.1) Required for native Markdown topic support
Git 2.34 Must be available on the system PATH
GitHub account Any Free accounts are sufficient; write access to the target repository is required
Text editor Any VS Code recommended for editing Markdown outside Flare

How this guide is organized

This guide is divided into six sequential sections. Complete them in order:

  1. Set up Git — Initialize a repository and configure the recommended .gitignore entries for a Flare project.
  2. Use Markdown in Flare — Create Markdown topic files, apply naming conventions, and add topics to the table of contents.
  3. Link Flare to Git — Bind the Flare project to a remote GitHub repository using Flare's built-in Source Control integration.
  4. Branching strategy — Create feature and hotfix branches to keep work-in-progress content separate from published content.
  5. Review and merge — Open pull requests, complete review checklists, resolve conflicts, and merge approved changes.
  6. Publish with CI/CD — Automate Flare builds and deploy the HTML5 WebHelp output using GitHub Actions.
