Publish documentation with CI/CD

Automate your MadCap Flare build and deployment by using a CI/CD pipeline. On every merge to main, the pipeline compiles the Flare project, validates the output, and deploys the HTML5 WebHelp to the developer portal.

Overview

MadCap Flare includes a command-line build tool, madbuild.exe, that compiles a Flare project without opening the Flare IDE. CI/CD platforms invoke madbuild.exe on a Windows build agent whenever a pull request is opened (to validate the build) or a merge to main completes (to deploy the output).

Note  madbuild.exe requires a licensed installation of MadCap Flare on the build agent. The license must be activated on the agent machine before the first build. Contact MadCap Software support for information about CI/CD licensing options.

Pipeline stages

The recommended pipeline includes the following stages in order:

Stage Trigger Action On failure
Validate Pull request opened or updated Compile the Flare project; report build errors as a PR status check Block merge; notify author
Lint Pull request opened or updated Run markdownlint on all changed .md files Block merge; list lint violations in PR comment
Link check Pull request opened or updated Scan the compiled output for broken hyperlinks Block merge; list broken URLs
Deploy Merge to main Upload the Output/ folder to the hosting environment Notify the team and roll back to the previous deployment

Run madbuild.exe

Use the following command syntax to compile a Flare project from the command line:

madbuild.exe -project "<path-to-project.flprj>" -target "<target-name>"

Example:

madbuild.exe -project "C:\repos\devdocs\DevDocs.flprj" -target "WebHelp"

madbuild.exe exits with code 0 on success and a non-zero code on failure. CI/CD platforms use the exit code to determine whether the build stage passes.

Parameter Description
-project Full path to the .flprj file
-target Name of the Flare build target as defined in the Target Editor
-log (Optional) Path to write the build log file

Configure GitHub Actions

The following YAML workflow file configures two jobs: a validate job that runs on every pull request, and a deploy job that runs on merges to main. 

name: Flare Build and Deploy

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  validate:
    runs-on: windows-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build Flare project
        run: |
          & "C:\Program Files\MadCap Software\MadCap Flare\Flare.app\madbuild.exe" `
            -project "${{ github.workspace }}\DevDocs.flprj" `
            -target "WebHelp"

  deploy:
    runs-on: windows-latest
    needs: validate
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      - name: Build Flare project
        run: |
          & "C:\Program Files\MadCap Software\MadCap Flare\Flare.app\madbuild.exe" `
            -project "${{ github.workspace }}\DevDocs.flprj" `
            -target "WebHelp"
      - name: Deploy to Azure Static Web Apps
        uses: Azure/static-web-apps-deploy@v1
        with:
          azure_static_web_apps_api_token: ${{ secrets.AZURE_STATIC_WEB_APPS_TOKEN }}
          repo_token: ${{ secrets.GITHUB_TOKEN }}
          action: upload
          app_location: Output/WebHelp

Deployment targets

The compiled Output/ folder contains static HTML, CSS, JavaScript, and media files that can be hosted on any static file server. Common deployment targets include:

  • Azure Static Web Apps — Use the Azure/static-web-apps-deploy GitHub Action or the Azure DevOps task.
  • Amazon S3 with CloudFront — Sync the output folder to an S3 bucket and invalidate the CloudFront cache after each deployment.
  • GitHub Pages — Push the output folder to the gh-pages branch by using the peaceiris/actions-gh-pages Action.
  • Self-hosted NGINX — Use SCP or rsync to transfer the output to the server document root.

Important  Store all deployment credentials—API tokens, connection strings, and access keys—as encrypted secrets in the CI/CD platform. Never hard-code credentials in workflow YAML files or commit them to the repository.
