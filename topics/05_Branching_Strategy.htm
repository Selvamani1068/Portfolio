Branching strategy for documentation projects

A consistent branching model ensures that documentation changes are reviewed before publication and that hotfixes can be applied independently of ongoing work. This topic describes the recommended branch structure for MadCap Flare documentation projects.

Overview

The recommended branching model is a simplified trunk-based workflow with short-lived feature branches. The main branch always represents the current published state of the documentation. No author commits directly to main—all changes enter through a pull request.

Admin note  Protect the main and release/* branches in the remote repository before inviting contributors. Require pull-request approval and a passing CI build before any merge is allowed.

Branch types and naming conventions

Branch type Naming pattern Branched from Merged into Purpose
Main main — — Published state of documentation; protected
Feature docs/<topic-name> main main New topics or significant content updates
Hotfix hotfix/<issue-id> main main Urgent corrections to published content
Release release/v<major.minor> main main Stabilization branch for a versioned output
Experiment exp/<idea-name> main Not merged (discard or cherry-pick) Template, structure, or tooling experiments

Feature branch workflow

Follow these steps to create and submit a documentation change:

  1. Confirm that your local main branch is up to date: 
    git checkout main
git pull origin main
  2. Create a feature branch by using the docs/ prefix: 
    git checkout -b docs/authentication-guide
  3. Author or edit the Markdown or XML topic files on the feature branch. Save all changes in Flare before committing.
  4. Stage and commit the changes with a descriptive message: 
    git add .
git commit -m "Add authentication guide topic and TOC entry"
  5. Push the feature branch to the remote repository: 
    git push -u origin docs/authentication-guide
  6. Open a pull request in the remote repository UI, targeting main. Complete the pull-request description and request a reviewer.
  7. After the pull request is approved and the CI build passes, merge the pull request.
  8. Delete the feature branch from the remote repository after the merge is complete: 
    git push origin --delete docs/authentication-guide

Important  Never commit directly to main. All changes must enter through a pull request to preserve a complete review history and to ensure the CI build validates the output before publication.

Hotfix workflow

Use a hotfix branch when you must correct a critical error in the published documentation without waiting for in-progress feature branches to complete:

  1. Branch from main: 
    git checkout main
git checkout -b hotfix/GH-142
  2. Make the correction, commit, and push.
  3. Open a pull request targeting main and mark it as a hotfix for expedited review.
  4. After approval and merge, delete the hotfix branch.

Commit message guidelines

Write commit messages in the imperative present tense. Follow this format:

<verb> <summary of change> [optional: issue reference]

Examples:
  Add API reference topic for authentication endpoints
  Fix broken cross-reference in branching strategy topic
  Update TOC to include new CI/CD publishing section  (closes #87)

Keep the summary line under 72 characters. Add a blank line before the body if additional context is needed.
