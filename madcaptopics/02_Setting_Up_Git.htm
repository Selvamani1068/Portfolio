Set up Git for your documentation project

Create a Git repository for your MadCap Flare project so that every Markdown and XML topic file is tracked under version control. This section covers installing Git, creating a repository on GitHub, initializing the local repository, and configuring the correct .gitignore entries.

Step 1: Verify your Git installation

Before you create a repository, confirm that Git is installed and available:

  1. Open a terminal (Command Prompt, PowerShell, or Git Bash on Windows; Terminal on macOS).
  2. Run the following command: 
    git --version
  3. Confirm that the output shows version 2.34 or later. For example: 
    git version 2.43.0.windows.1

Note  If Git is not installed, download it from git-scm.com/downloads and run the installer. Accept the default options. After installation, restart your terminal and repeat step 2.

Step 2: Configure your Git identity

Git attaches your name and email address to every commit you create. Set these values once on each machine you use:

  1. Set your display name: 
    git config --global user.name "Your Name"
  2. Set your email address (use the same address as your GitHub account): 
    git config --global user.email "you@example.com"
  3. Verify the configuration: 
    git config --global --list
    The output should include user.name and user.email.

Step 3: Create a repository on GitHub

Create an empty remote repository before you initialize your local project:

  1. Sign in to github.com.
  2. In the upper-right corner, select +, and then select New repository.
  3. In the Repository name field, enter a name for your documentation project. For example: developer-docs.
  4. Set the visibility to Public or Private according to your project requirements.
  5. Leave the Initialize this repository options blank — do not add a README, .gitignore, or license at this step. You will create these files locally.
  6. Select Create repository.
  7. On the Quick setup page, copy the HTTPS URL shown in the setup box. It looks like this: 
    https://github.com/your-username/developer-docs.git
    You will use this URL in Step 5.

Step 4: Create the .gitignore file

A .gitignore file tells Git which files to exclude from version control. For a Flare project, exclude build output and temporary files:

  1. Open your Flare project root folder — the folder that contains the .flprj file.
  2. Create a new file named .gitignore (no file extension).
  3. Add the following entries to the file: 
    # MadCap Flare build output
Output/

# Flare temporary and backup files
*.mcbackup
*.flbkp
*.bak
.mclog
  4. Save and close the file.

Important  Never commit the Output/ folder to the repository. The build output is regenerated by the CI/CD pipeline on every merge and does not belong in version history.

The following table explains each entry:

Entry What it excludes
Output/ All compiled HTML, PDF, and WebHelp build output
*.mcbackup Automatic Flare topic backup files
*.flbkp Flare project snapshot backup files
*.bak Generic editor backup files
.mclog Flare session and build log files

Step 5: Initialize the local repository and push to GitHub

Initialize a Git repository in your Flare project root folder and upload it to GitHub:

  1. In your terminal, navigate to the Flare project root folder: 
    cd path/to/your/flare-project
  2. Initialize the Git repository: 
    git init
  3. Rename the default branch to main: 
    git branch -M main
  4. Stage all project files for the initial commit: 
    git add .
  5. Create the initial commit: 
    git commit -m "Initial commit: MadCap Flare documentation project"
  6. Connect your local repository to the GitHub remote. Replace the URL with the one you copied in Step 3: 
    git remote add origin https://github.com/your-username/developer-docs.git
  7. Push the initial commit and set the upstream tracking branch: 
    git push -u origin main

Note  When you push for the first time, your browser may open a GitHub sign-in page to authenticate. Complete the sign-in, and then return to your terminal. The push will complete automatically.

Step 6: Verify the setup

Confirm that the repository is initialized and connected correctly:

  1. Check the working tree status: 
    git status
    The output should display: nothing to commit, working tree clean
  2. View the commit log: 
    git log --oneline
    Your initial commit message should appear in the list.
  3. Confirm the remote is configured: 
    git remote -v
    The output should show your GitHub URL under both fetch and push.
  4. Refresh your repository page on GitHub. Your project files should be visible.
