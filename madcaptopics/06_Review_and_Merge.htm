Review and merge documentation changes

Every documentation change must be reviewed and approved through a GitHub pull request before it is merged into main. This section shows you how to prepare your changes for review, open a pull request, conduct a review, resolve merge conflicts, and complete the merge.

Step 1: Complete the pre-review checklist before opening a pull request

Address the following items before you request a review. Resolving these issues before review reduces the number of revision cycles.

Check How to verify
The Flare project builds without errors In Flare, select Build > Build Primary and open the Build Log. Confirm there are no errors.
All changed .md files are valid Markdown Run markdownlint *.md in the topics folder. Resolve any lint violations before pushing.
New topics are added to the TOC Open the .fltoc file in Flare's TOC Editor and confirm the new topic node is present in the correct position.
No broken cross-references In Flare, select Tools > Project Analysis > Analyzer and check the Broken Links section.
Content follows the Microsoft Writing Style Guide Review headings and step text for imperative mood, second person (you), and active voice. Check UI element names are bold.
Commit messages follow the project convention See Write commit messages for the required format.

Step 2: Open a pull request on GitHub

  1. Go to your repository on github.com.
  2. GitHub displays a banner: Your branch had recent pushes. Select Compare & pull request. Alternatively, select the Pull requests tab and then select New pull request.
  3. Confirm the base branch is set to main and the compare branch is your feature branch.
  4. Enter a clear pull request title. Use the same imperative format as commit messages. For example: Add authentication guide topic.
  5. In the description field, include:
    • A one-sentence summary of what changed and why.
    • A link to the related issue or task (for example: Closes #45).
    • A note about any files that reviewers should focus on.
  6. In the Reviewers field, assign at least one reviewer.
  7. Select Create pull request.

Step 3: Review a pull request (reviewer steps)

As a reviewer, complete the following checks before you approve:

  1. In the pull request, select the Files changed tab. Read through all changes in the diff view.
  2. Verify that:
    • Content is factually accurate and complete.
    • Headings and steps follow the Microsoft Writing Style Guide (imperative mood, active voice, second person).
    • Code samples use the correct fenced code block language identifier (for example, ```bash or ```yaml).
    • New topics are present in the TOC change (.fltoc file diff).
    • No credentials, API keys, or internal URLs appear in the content.
  3. To leave feedback on a specific line, select the + icon next to the line number and type your comment. Select Start a review.
  4. When you finish reviewing all files, select Review changes in the upper-right corner of the Files changed tab.
  5. Select Approve if the changes are ready to merge, or Request changes if revisions are needed.
  6. Select Submit review.

Note  After the author addresses your comments, mark each comment thread as Resolved so the author can track outstanding feedback.

Step 4: Resolve merge conflicts

Merge conflicts occur when two branches modify the same file in incompatible ways. If GitHub reports a conflict, resolve it before the pull request can be merged:

  1. In your terminal, check out your feature branch and fetch the latest main: 
    git checkout docs/your-feature-branch
git fetch origin
git merge origin/main
  2. Open each file that Git reports as conflicted. Git marks the conflicting sections: 
    <<<<<<< HEAD
  (your changes on the feature branch)
=======
  (incoming changes from main)
>>>>>>> origin/main
  3. Edit the file to keep the correct content. Remove all conflict markers (<<<, ===, >>>) completely.
  4. Stage the resolved file: 
    git add path/to/resolved-file.htm
  5. Complete the merge commit: 
    git commit -m "Merge main into docs/your-feature-branch; resolve conflict in resolved-file.htm"
  6. Push the updated branch: 
    git push
  7. Return to the pull request on GitHub. The conflict warning should be resolved.

Important  Flare .fltoc and .flprj files are XML. Merge conflicts in these files can corrupt the project structure. Coordinate with your project administrator before making structural changes to these files, and review resolved conflicts carefully in the Flare IDE.

Step 5: Merge and clean up

  1. Confirm that all reviewer-requested changes are addressed and all status checks pass (look for the green check marks on the pull request page).
  2. Select Merge pull request.
  3. Select Squash and merge for feature branches. This combines all commits from the branch into a single commit on main, keeping the commit history clean.
  4. Enter a final commit message for the squashed commit and select Confirm squash and merge.
  5. Select Delete branch to remove the feature branch from GitHub.
  6. Update your local repository: 
    git checkout main
git pull origin main
git branch -d docs/your-feature-branch
