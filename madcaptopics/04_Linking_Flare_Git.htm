Connect your Flare project to GitHub

Connect MadCap Flare's built-in Source Control integration to your GitHub repository. After you complete this setup, you can commit, push, pull, and create branches without leaving the Flare IDE.

Before you begin

Confirm the following before you start:

  • You have completed Set up Git for your documentation project.
  • Git 2.34 or later is installed and available on the system PATH.
  • The Flare project has been saved to a local folder.
  • The remote repository exists on GitHub and you have write access to it.
  • The local project folder does not already contain a .git directory (if it does, skip to Verify the connection).

Step 1: Open the Bind Project dialog

  1. Open your Flare project.
  2. On the menu bar, select Source Control.
  3. Select Bind Project. The Bind Project dialog opens.

Important  Do not use Bind Project if the project folder already contains a .git directory. Binding overwrites the existing Git configuration. Instead, select Source Control > Get Latest to connect to the existing repository.

Step 2: Configure the Source Control binding

  1. In the Source Control dropdown, select Git.
  2. In the Remote field, enter the HTTPS URL of your GitHub repository. For example: 
    https://github.com/your-username/developer-docs.git
  3. Enter your GitHub credentials:
    • Username: your GitHub username.
    • Password: your GitHub Personal Access Token (PAT). GitHub no longer accepts account passwords for Git operations. To create a PAT, go to GitHub > Settings > Developer settings > Personal access tokens > Generate new token and select the repo scope.
  4. Select OK.

Note  Flare initializes a local .git repository in the project root, stages all project files, creates an initial commit, and pushes it to the remote. This may take a minute if the project is large.

Step 3: Push the initial commit

  1. When Flare displays the prompt Do you want to push to the remote repository?, select Push.
  2. Flare uploads the initial commit to GitHub. When the operation completes, the status bar at the bottom of the Flare window shows the current branch name (usually main).

Step 4: Verify the connection

  1. On the menu bar, select Source Control > View History. The initial commit should appear in the history list with your name, email, and the timestamp.
  2. Open any topic file and add a space at the end of a line. Save the file. The file icon in the Content Explorer should display a pending-change indicator (a small pencil or checkmark, depending on your Flare version).
  3. Select Source Control > Check In. Enter a test commit message and select Check In. The indicator disappears.
  4. Select Source Control > View History again. Your test commit should appear in the list.
  5. Go to your repository on GitHub and refresh the page. Your files and the latest commit should be visible.

Daily workflow reference

The following table maps everyday tasks to their Flare menu path and the equivalent Git command in the terminal:

Task Flare menu path Git command
Get the latest changes from GitHub Source Control > Get Latest git pull
Commit your changes Source Control > Check In git add . && git commit -m "message"
Push commits to GitHub Source Control > Push git push
Create a new branch Source Control > Branch > Create Branch git checkout -b branch-name
Switch to another branch Source Control > Branch > Switch Branch git checkout branch-name
View commit history Source Control > View History git log --oneline
Discard unsaved changes to a file Source Control > Undo Check Out git checkout -- filename

Troubleshoot common issues

Symptom Likely cause Resolution
Flare cannot detect Git Git is not on the system PATH Add the Git binary folder to your system PATH environment variable and restart Flare
Push fails with authentication error Personal Access Token is expired or has insufficient scope Generate a new PAT with the repo scope on GitHub and update your credentials in Flare
Source Control menu items are grayed out The project is not bound to source control Select Source Control > Bind Project and complete the steps in this topic
Push rejected: remote contains work not in local Someone else pushed to the branch since your last pull Select Source Control > Get Latest to pull, resolve any conflicts, and push again
