Manage documentation changes with branches

Use Git branches to keep work-in-progress content separate from your published documentation. This section explains the branch structure, shows you how to create a branch for a new topic, and describes when to use a hotfix branch.

Branch structure overview

The main branch always reflects the currently published state of the documentation. No author commits directly to main. All changes enter through a pull request from a short-lived branch.

Branch type Naming pattern Branched from Purpose
Main main — Published state; protected. No direct commits.
Feature (new topic or major update) docs/topic-name main New topics or significant content updates
Hotfix (urgent correction) hotfix/issue-id main Fixing a critical error in published content
Release (versioned output) release/v1.2 main Stabilization branch for a versioned documentation release

Create a feature branch for a new topic

Follow these steps every time you start a new documentation task — a new topic, a content update, or a structural change:

  1. Make sure your local main branch is up to date: 
    git checkout main
git pull origin main
  2. Create a new feature branch. Use the docs/ prefix followed by a short description in kebab-case: 
    git checkout -b docs/authentication-guide
  3. Open Flare. In Source Control > Branch > Switch Branch, confirm that the new branch is selected.
  4. Create or edit your Markdown or XML topic files on this branch. Save all changes in Flare before you commit.
  5. Stage your changes: 
    git add .
  6. Write a commit message. Use the imperative present tense and keep the summary under 72 characters: 
    git commit -m "Add authentication guide topic and TOC entry"
  7. Push the feature branch to GitHub: 
    git push -u origin docs/authentication-guide
  8. Open a pull request on GitHub (see Review and merge documentation changes).

Write commit messages

Good commit messages make the change history readable and help reviewers understand what changed and why. Follow this format: 

<imperative verb> <brief summary> [optional issue reference]
Good example Why it works
Add API reference topic for authentication endpoints Starts with an imperative verb; describes what was added
Fix broken cross-reference in branching strategy topic Starts with a verb; identifies the file and the problem
Update TOC to include CI/CD publishing section (closes #87) Describes the change and links to the issue that requested it
Poor example Why to avoid it
updates Vague; does not describe what changed
fixed stuff Does not use the imperative; gives no context
WIP Not a complete commit message; push a draft PR instead

Fix an urgent error with a hotfix branch

Use a hotfix branch when you must correct a critical error in the published documentation immediately, without waiting for in-progress feature branches to be ready:

  1. Check out main and make sure it is current: 
    git checkout main
git pull origin main
  2. Create a hotfix branch. Use the issue number or a short description: 
    git checkout -b hotfix/incorrect-api-endpoint
  3. Make the correction in Flare, save the file, and run a local build to verify.
  4. Stage and commit: 
    git add .
git commit -m "Fix incorrect API endpoint URL in authentication guide (closes #142)"
  5. Push the hotfix branch: 
    git push -u origin hotfix/incorrect-api-endpoint
  6. Open a pull request on GitHub targeting main. In the PR description, mark it as a hotfix so reviewers can prioritize it.
  7. After the PR is approved and merged, delete the hotfix branch from GitHub.

Important  Never commit directly to main, even for hotfixes. All changes must enter through a pull request so that the CI build validates the Flare output before it is published.
