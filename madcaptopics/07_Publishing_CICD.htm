Automate builds and publish documentation with GitHub Actions

Set up a GitHub Actions workflow that automatically compiles your MadCap Flare project and publishes the HTML5 WebHelp output whenever a change is merged into main. The pipeline also validates the build on every pull request, so broken builds are caught before they reach your published documentation.

How the pipeline works

MadCap Flare includes a command-line build tool called madbuild.exe that compiles a Flare project without opening the IDE. GitHub Actions runs madbuild.exe on a Windows virtual machine every time you push to a pull request or merge to main.

Stage Trigger What it does On failure
Validate Pull request opened or updated Compiles the Flare project; reports errors as a required status check on the PR Blocks the merge; notifies the author
Lint Pull request opened or updated Runs markdownlint on all changed .md files Blocks the merge; lists lint violations in a PR comment
Deploy Merge to main Uploads the compiled Output/ folder to GitHub Pages Notifies the team; the previous deployment remains live

Note  madbuild.exe requires a licensed installation of MadCap Flare on the build machine. The license must be activated on the GitHub Actions runner before the first build. Contact MadCap Software support for information about CI/CD licensing options.

Step 1: Create the GitHub Actions workflow file

  1. In your repository root, create the folder path .github/workflows/ if it does not already exist.
  2. Create a new file named flare-build.yml inside that folder.
  3. Paste the following YAML into the file: 
    name: Flare Build and Deploy

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  validate:
    runs-on: windows-latest
    steps:
      - name: Check out repository
        uses: actions/checkout@v4

      - name: Build Flare project (validate)
        run: |
          & "C:\Program Files\MadCap Software\MadCap Flare\Flare.app\madbuild.exe" `
            -project "${{ github.workspace }}\DevDocs.flprj" `
            -target "WebHelp"

  deploy:
    runs-on: windows-latest
    needs: validate
    if: github.ref == 'refs/heads/main'
    steps:
      - name: Check out repository
        uses: actions/checkout@v4

      - name: Build Flare project (deploy)
        run: |
          & "C:\Program Files\MadCap Software\MadCap Flare\Flare.app\madbuild.exe" `
            -project "${{ github.workspace }}\DevDocs.flprj" `
            -target "WebHelp"

      - name: Deploy to GitHub Pages
        uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./Output/WebHelp
  4. In the -project parameter, replace DevDocs.flprj with the actual name of your .flprj file.
  5. Save the file.

Step 2: Enable GitHub Pages for deployment

  1. Go to your repository on GitHub.
  2. Select Settings.
  3. In the left sidebar, select Pages.
  4. Under Build and deployment, set Source to Deploy from a branch.
  5. Set Branch to gh-pages and the folder to / (root).
  6. Select Save.

Note  The gh-pages branch is created automatically by the peaceiris/actions-gh-pages action the first time the deploy job runs. You do not need to create it manually.

Step 3: Push the workflow and trigger the first build

  1. Stage and commit the workflow file: 
    git add .github/workflows/flare-build.yml
git commit -m "Add GitHub Actions workflow for Flare build and deploy"
  2. Push to a feature branch and open a pull request to trigger the validate job: 
    git push -u origin docs/add-cicd-workflow
  3. On the pull request page on GitHub, select the Checks tab. Watch the validate job. A green check mark means the Flare build succeeded.
  4. After the pull request is approved and merged, the deploy job runs automatically and publishes the output to GitHub Pages.

Step 4: Verify the published output

  1. After the deploy job completes, go to: 
    https://your-username.github.io/your-repo-name/
  2. Confirm that the HTML5 WebHelp output loads correctly and that the table of contents, topic content, and navigation all work as expected.
  3. To view the deploy history, go to your repository and select the Actions tab. Each workflow run shows the commit that triggered it and whether the build and deploy succeeded.

madbuild.exe parameter reference

Parameter Description Example
-project Full path to the .flprj file -project "C:\repos\docs\DevDocs.flprj"
-target Name of the build target as defined in the Flare Target Editor -target "WebHelp"
-log (Optional) Path to write the build log file -log "C:\logs\build.log"

madbuild.exe exits with code 0 on success and a non-zero exit code on failure. GitHub Actions uses the exit code to determine whether the build step passes or fails.

Important  Store all deployment credentials — API tokens, connection strings, and access keys — as encrypted secrets in your GitHub repository settings (Settings > Secrets and variables > Actions). Never hard-code credentials in workflow YAML files or commit them to the repository.
